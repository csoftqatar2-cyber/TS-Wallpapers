## 1. What this project is

**TS Wallpapers** (Arabic: ذبذبة خلفيات, publisher brand "THABTHABA STORE") is a
commercial Android wallpaper/screensaver app sold outside app stores, plus its
management backend. It is a GPLv3 fork of
[FsClock-Android](https://github.com/schorschii/FsClock-Android) by Georg Sieber
(fullscreen clock/screensaver), extended with:

- a wallpaper slideshow engine (image / GIF / MP4) fed from a Supabase backend,
- a per-device **activation** system (serial numbers, hardware IDs, block/unblock),
- per-device *and* global wallpaper assignment managed from a web dashboard,
- an in-app **self-update** system (APK download + install, no app store),
- LAN phone-pairing upload (QR code → local HTTP server on the TV/device).

There are **three subsystems** in one repo plus one CI pipeline:

| Subsystem | Where | Tech |
|---|---|---|
| Android app | `source/` | Java, Gradle, single module `:app`, 3 flavors |
| Admin web dashboard | `wallpapers_manager.html` (repo root) | Single-file HTML/JS, deployed to Vercel |
| Supabase backend | project `ihgmqwzdpugdzddobhbc` (remote) | Postgres + Storage + Auth + 1 Edge Function |
| CI publish pipeline | `.github/workflows/release.yml` | GitHub Actions → builds APK → uploads to Supabase |

The Supabase backend is **only partially captured in this repo** (see §6 — this is
the single most important thing to know before touching backend behavior).

## 2. Data flows (end to end)

### Wallpaper delivery (device ← Supabase)
1. App derives a hardware id (`WallpaperRepo.getHardwareId`, prefixes `VIN-`/`MAC-`/`SYS-`/`BOOT-`/`SRL-`/`AID-`).
2. App POSTs to Supabase RPC `get_wallpapers` with `{device_hw_id, legacy_hw_id}` →
   receives a device-specific playlist (global rows + rows matching its hardware_id).
   A single-item response `"inactive"` means "device not activated" and clears the playlist.
3. Playlist is cached in SharedPreferences (`wallpaper-cache-json`); videos are
   pre-downloaded to `cacheDir/wallpapers/vid_*.bin`. Re-sync every 5 minutes.
4. Local files in `externalFilesDir/Wallpapers/` (file picker or LAN upload) are merged in.

### Activation (device → Supabase)
- User enters a serial (must start with `7078` — enforced client-side in
  `FsClockView.java:224` and `WallpaperRepo.java:224`) → POST RPC `activate_device` →
  on `"success"`, `device-active=true` is stored via `SecurePrefs` (HMAC-signed,
  keyed off the APK signing cert — re-signing the APK invalidates activation).

### Admin management (dashboard → Supabase)
- `wallpapers_manager.html` (deployed at Vercel, `/` rewrites to it) logs in as the
  fixed Supabase Auth user `admin@tswallpapers.app` (password → JWT).
- Wallpaper **file uploads go through Edge Function `admin-upload`** (service-role
  write to the `wallpapers` storage bucket); metadata rows go to the `wallpapers` table.
- Device admin (block/rename/activate/delete) PATCHes the `devices` table.

### Self-update (CI → Supabase → devices)
1. Bump `versionCode`/`versionName` in `source/app/build.gradle`, push to `main`.
2. `release.yml` reads the version; **gate:** publishes only if `versionCode` > the
   highest in the `app_versions` table. Builds `:app:assembleStandaloneRelease`,
   signed with `source/thabthaba.jks` (from GitHub secret).
3. Uploads APK to the public `apk` storage bucket; inserts a row in `app_versions`
   (changelog = last commit subject line — **commit messages on version bumps are user-visible**).
4. Devices poll `app_versions` (`UpdateManager`); newer `version_code` → in-app
   update badge → DownloadManager + package installer.

## 4. Identity and versioning facts

- Published application id: **`store.thabthaba.clock`**. Java namespace/package:
  **`systems.sieber.fsclock`** (upstream's — intentionally kept; do not "fix" the mismatch).
- App name: "TS Wallpapers" (`values/strings.xml`), Arabic "ذبذبة خلفيات" (`values-ar`).
- Current version: `versionCode 118`, `versionName "1.4"` (`source/app/build.gradle:19-20`).
- **Versioning contract:** `versionCode` is globally monotonic and must stay above 113
  (the last upstream-lineage build "2.3-ts"). versionName was reset to 1.x for the fork.
  Every release = bump both, push to main; CI does the rest. Never reuse or lower a versionCode.
- License: **GPLv3** (`source/LICENSE.txt`). The fork is a conveyed modified work:
  keep it GPLv3, keep modification notices, source must be available to APK recipients.
  DSEG fonts are SIL OFL (`source/app/src/main/res/raw/dseg_license.txt`) — keep attribution.

## 6. Supabase backend — committed vs live-only ⚠️

The live Supabase project contains objects that exist **nowhere in this repo**.
Rebuilding the backend from the repo alone is NOT possible. Committed SQL
(`supabase_app_update_setup.sql`) creates only: `app_versions` table (+RLS, anon-writable)
and the public `apk` bucket (+anon policies).

**Live-only (undocumented, inferred from client code):**
- `wallpapers` table — columns `id, url, type ('image'|'video'), is_global, hardware_id, created_at` + its RLS
- `devices` table — columns `hardware_id, serial_number, client_name, is_active, is_blocked, created_at` + its RLS
- `wallpapers` storage bucket + object policies
- RPC functions `get_wallpapers(device_hw_id, legacy_hw_id)` and
  `activate_device(...)` (returns `"success"|"serial_already_used"|"invalid_format"|"blocked"`; playlist sentinel `"inactive"`)
- Edge Function **`admin-upload`** (service-role storage writes; checks admin JWT)
- Auth user `admin@tswallpapers.app`
- Email-scoped RLS write lock on admin-writable tables (per project memory)

**Server contract the app depends on** (breaking any of these bricks fielded devices):
RPC names and response shapes above; `app_versions` columns
`version_code, version_name, apk_url, changelog`; hardware-id prefix format;
serial prefix `7078`.

The Supabase MCP server is usually connected in this workspace — use it
(`list_tables`, `execute_sql`, `get_advisors`) to inspect the live schema instead of guessing.

## 7. Global editing invariants (violating these breaks production)

1. **Paired layouts:** `res/layout/view_fsclock.xml` and `res/layout-land/view_fsclock.xml`
   must be edited in tandem — identical view-ID sets; `FsClockView` findViewById's them and
   NPEs on rotation if one diverges. (Full ID list in the res/ README.)
2. **SharedPreferences domain is `"CLOCK"`** and pref key strings are contracts shared by
   FsClockView, widgets, WallpaperRepo, UploadServer, and `FsClockApp.migrateSettings()`.
3. **ProGuard:** any new XML-referenced custom view, Gson model, or reflection target
   needs a keep rule in `source/app/proguard-rules.pro`. `Log.v/d/i/w` are stripped in
   release — never gate logic on them; `Log.e` survives.
4. **StringObfuscator is the single source of the Supabase URL/key** for the app. If the
   URL ever decodes to something containing `YOUR_SUPABASE_PROJECT`, activation silently
   fail-opens and sync/update no-op.
5. **SecurePrefs HMAC is signing-cert-bound.** Changing `deriveKey`/HMAC format or the
   signing key invalidates activation on every installed device.
6. **CI publishes on every push to main** where versionCode was bumped. Pushing a version
   bump = releasing to all devices. The commit subject becomes the user-visible changelog.
7. **Do not run** `supabase_app_update_setup.sql` against the live project casually — it
   drops/recreates policies.
8. Gradle build path must contain **no spaces** when building locally per the Arabic guide
   (this OneDrive path `TS WALLPAPERS` has spaces — CI builds are the reliable path).

## 8. Known issues / flags (found during audit — intentionally NOT fixed)

### Functional / correctness
- **Data race:** `WallpaperRepo.mItems`/`mIndex` mutated on background sync threads, read
  on UI thread, no synchronization (`WallpaperRepo.java:363` et al.). Overlapping syncs possible.
- **Widget default mismatch:** digital widget defaults 24h=true (`FsClockWidgetDigitalProvider.java:71`)
  vs main clock 24h=false (`FsClockView.java:625`, `BaseSettingsActivity.java:340`).
- **`checkCode` bug:** `BaseSettingsActivity.java:1652-1654` reads `getErrorStream()` after
  `disconnect()`; also treats HTTP 999 as success.
- **Notification count unreliable:** `FsClockView.java:1314` adds `-1` when the `count`
  extra is missing; listener uses `Notification.number` which is usually 0.
- **JSON via string concat** with unescaped input in `WallpaperRepo.java:523,556` (activation body).
- **`.mkv` misclassified:** manager allows `.mkv` (`wallpapers_manager.html:1906`) but only
  `.mp4`/`.bin` are typed `video` (`:2135`) → `.mkv` rows get `type=image`. Same for GIF:
  manifest format has type `gif`, manager only ever writes `image`/`video`.
- **`app_versions.mandatory`** column exists but nothing reads or writes it.

### Security-theater (looks like protection, isn't)
- **IntegrityGuard signature check is a no-op:** `sExpectedSigHash` is never baked in;
  `init()` trusts whatever cert is present at first run (`IntegrityGuard.java:41-60`).
- **StringObfuscator is trivially reversible** (XOR key co-located, `StringObfuscator.java:17-63`).
  Fine for the anon key, but do not put real secrets through it.
- **Serial/unlock gate is client-side only:** any 6-digit code starting `7078` unlocks the
  standalone build (`standalone/SettingsActivity.java:83`); same prefix hardcoded twice more.
- **`app_versions` + `apk` bucket are anon-WRITABLE** (`supabase_app_update_setup.sql:34-38,53-58`):
  anyone with the public anon key could publish a malicious "update" row/APK. The SQL's
  comment claims this "matches the wallpapers/devices model," but the dashboard actually
  uses an admin JWT — the two security narratives conflict. Highest-impact flag in the repo.
- **Cleartext traffic enabled globally** (`AndroidManifest.xml:23`) for `http://ip-api.com` etc.

### Stale / dead / cosmetic
- `source/fastlane/metadata/` (all 8 locales) still describes upstream FsClock, not this app.
- `source/README.upstream.md` is the verbatim upstream README (kept for attribution; renamed
  from README.md during reorganization so it can't be mistaken for fork docs).
- Dead code: `FullscreenActivity.showDialogReview()` (`FullscreenActivity.java:372-401`),
  review-counter machinery, commented subscription branch (`google/FeatureCheck.java:127-135`),
  upstream app cross-promo remnants (`BaseSettingsActivity.java:1484-1488`).
- Amazon IAP receiver declared in the **main** manifest for all flavors (`AndroidManifest.xml:86`).
- Version numbers in docs disagree (SQL example says 2.4/43; Arabic guide says v1.2; real is 1.4/118) —
  the gradle file is the only source of truth.
- Repo-owner mismatch worth verifying: `setup-ci-secrets.sh:15` targets `csoftqatar2-cyber/TS-Wallpapers`;
  git user is `csoftqatar2-cyber`, workspace login is csoftqatar4@gmail.com.
- `tools/ForBiggerBlazes.mp4` is a 706-byte stub, not a playable video.
- Arabic UI strings are largely hardcoded in default `values/strings.xml` and inline in
  layouts (activation overlay), so `values-ar` only overrides 13 of 162 strings.
